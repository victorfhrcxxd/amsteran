## PHASE 1 — PROJECT ARCHITECTURE OVERVIEW

### Base Framework
- **aCis** (L2J fork, Java 8) — standard aCis Interlude/C4-era server
- Eclipse project, Ant build (`build.xml`)

## PHASE 2 — FEATURE MAPPING

### Autofarm (Online)
- **Starts**: `VoicedAutoFarm.useVoicedCommand("farm_on")` → `AutofarmManager.INSTANCE.startFarm(player)`
- **Manager**: `AutofarmManager` (enum singleton), tick = 450ms via `ThreadPoolManager`
- **Routine**: `AutofarmPlayerRoutine.executeRoutine()` — targets monsters, uses skills/potions, checks buffs
- **Config**: `Config.AUTOFARM_*` (loaded from `l2jmod.properties`)
- **SQL**: None
- **Dependencies**: `OfflineFarmManager.isInActiveEvent()`, `VoicedAutoFarm.showAutoFarm()`

### Offline Farm
- **Starts**: `VoicedOfflineFarm` → `OfflineFarmManager`
- **Manager**: `OfflineFarmManager` (standard singleton)
- **Config**: `Config.ENABLE_OFFLINE_FARM`, `Config.OFFLINE_FARM_*`
- **SQL**: `offline_farm_time` table
- **Dependencies**: All event systems (TvT/CTF/DM/LM/KTB checked in `isInActiveEvent()`)

### VIP System
- **Starts**: `VoicedVipManager` / admin command
- **Manager**: `VipRewardManager` — **body is completely commented out**
- **Config**: `l2jmod.properties` VIP section (EnableVipSystem, VIPXpSpRate, etc.)
- **SQL**: `reward_vip_manager` table (**SQL script missing**)
- **Status**: ⚠️ Partially non-functional

### HWID / IP Protection
- **Starts**: Login hook / `L2PcInstance` enter world
- **Managers**: `HwidManager.validBox()`, `IPManager.validBox()`
- **Config**: `Config.HWID_MULTIBOX_PROTECTION_CLIENTS_PER_PC`, `Config.MULTIBOX_PROTECTION_CLIENTS_PER_PC`
- **SQL**: None (no persistence)
- **External**: `hwid.Hwid` JAR

### PvP Events (TvT / CTF / DM / LM / KTB / FOS)
- **Starts**: Scheduled by each `*Manager` on server boot; or manually via admin command
- **Config**: Each event has its own `*Config.init()` + `config/events/*.properties`
- **SQL**: None for most events (state is in-memory)
- **Pattern**: Each event follows `Config → Manager → Event state class → Teleporter NPC`

### Daily Reward
- **Starts**: `DailyRewardManager.claimDailyReward(player)` triggered from `DailyRewardCBManager`
- **Manager**: `DailyRewardManager`
- **Config**: `Config.ALLOW_DAILY_REWARD`, `Config.DAILY_LOG_REWARDS`
- **SQL**: `reward_manager` table (**SQL script missing**)
- **Bug**: Double check on `checkForLatestHWIDReward()` (see issue #1)

### Balance System
- **Starts**: Loaded at boot by `BalanceManager`
- **Manager**: `BalanceManager` — DB-backed class damage modifiers
- **Config**: None (pure DB)
- **SQL**: `balance` table (**SQL script missing**)
- **Admin**: `AdminBalance` command

### DressMe System
- **Starts**: `DressMeData.getInstance()` at boot, player interaction via `VoicedDressMe`
- **Manager**: `DressMeData` (datatable), `DressMeBypassHandler`
- **Config**: `Config.ALLOW_DRESS_ME`, `Config.DRESS_ME_*`
- **SQL**: `characters_dressme_data` ✓
- **Data**: `data/xml/dressme.xml`

### Ranking Systems
- **TopKillerRoundSystem**: Cycle-based PvP/PK tracker
- **CharacterKillingManager**: Per-cycle kill leader with `character_kills_info` DB table (**SQL missing**)
- **VoicedRanking**: Shows rankings via HTML
- **ClanRankingManager**: Clan-level score tracking

## PHASE 3 — PROPOSED CODE ORGANIZATION

Current structure mixes core aCis code with all custom code in the same packages.  
**Proposed** (non-breaking — rename packages, no functional change):

```
net.sf.l2j.gameserver
├── core/            ← Unchanged aCis core (no custom code)
├── custom/
│   ├── managers/    ← Move: instancemanager/custom/* here
│   ├── autofarm/    ← Move: instancemanager/autofarm/* here
│   ├── events/      ← Move: model/entity/events/* here
│   ├── datatables/  ← Move: datatables/custom/* here
│   └── handlers/    ← Move: custom voiced/admin handlers here
```

> ⚠️ **NOTE**: Package moves require updating all import statements.  
> This should only be done with an automated refactoring tool (Eclipse Refactor > Move).  
> Manual move is NOT recommended.

## PHASE 5 — MANAGER ISSUES (Code-Level)

### DailyRewardManager — CRITICAL BUG
```java
// Lines 80-83: checkForLatestHWIDReward called TWICE with same args
if (checkForLatestHWIDReward(player, "hwid")) // Rewarded
{
    if (checkForLatestHWIDReward(player, "hwid")) // REDUNDANT — always true here
        player.sendMessage("...");
}
```
**Fix**: Remove inner duplicate call.

### DailyRewardManager — INVALID CALENDAR TIME
```java
String[] time = "24:00".split(":");             // HARDCODED
c.set(Calendar.HOUR_OF_DAY, Integer.parseInt(time[0])); // HOUR_OF_DAY=24 → rolls to next day, but
c.set(Calendar.DAY_OF_MONTH, c.get(Calendar.DAY_OF_MONTH)+1); // already +1 day → skips a day
```
**Fix**: Use `HOUR_OF_DAY = 0` of `DAY_OF_MONTH + 1` consistently.

### ZergManager — NPE RISK
```java
String zerg1 = activeChar.getClan().getAllyName(); // NPE if clan is null
String zerg2 = player.getClan().getAllyName();     // NPE if player clan is null
```
**Fix**: Add null checks before accessing clan.

### AutofarmManager — ENUM PATTERN + EARLY TASK START
- Uses `enum` singleton instead of `SingletonHolder` — inconsistent with codebase
- `onUpdateTask` initialized at field declaration (class load time) → runs before server is ready
**Fix**: Move task start to explicit `init()` method.

### VipRewardManager — DEAD CODE
- Constructor's config check and entire `claimVipReward()` are commented out
- Still instantiated in `GameServer.java`
**Fix**: Either re-enable or stub cleanly.

## PHASE 6 — EVENT SYSTEM ISSUES

1. **No base interface/class** for events — TvT, CTF, DM, LM all re-implement start/stop/register/unregister independently
2. **No unified reward interface** — each event distributes rewards differently
3. **State consistency** — event state (started/stopped/waiting) is managed via static fields + enums inconsistently
4. **No guard against double-start** — if a manager's scheduled task fires while a previous event is still running, behavior is undefined
5. **Player cleanup on event end** — OfflineFarm players are ejected from events, but resurrection state and buff restoration differ per event

---

## PHASE 7 — PERFORMANCE ISSUES

1. `ZergManager.checkAllyAreaKickTask()` iterates ALL online players on every zone-enter check
2. `DailyRewardManager.Cd()` opens a new DB connection per call — should cache per player session
3. `AutofarmPlayerRoutine` calls `getAllEffects()` every 450ms tick for buff check — expensive
4. `StartupSystem.java` (94KB) likely has multiple responsibility overloads; loads heavy data synchronously
5. `BalanceManager.getMod()` iterates all `ClassId.values()` twice in worst case

---

## PHASE 8 — SECURITY ISSUES

1. **ZergManager NPE** (see Phase 5) — can crash thread if player without clan enters NO_ZERG zone
2. **IPManager** — `getClient().getConnection().getInetAddress()` — no null check on client or connection
3. **DailyRewardManager** — HWID can be null if client spoofs connection; no null guard before `player.getHWid()`
4. **VoicedAutoFarm** — no check if player is already in event before starting autofarm (OfflineFarmManager handles it, but AutofarmManager does not)
5. **AdminBalance** — balance modifications have no audit logging
6. **`GmNames` config** — fixed GM name in config file. If leaked, allows social engineering
